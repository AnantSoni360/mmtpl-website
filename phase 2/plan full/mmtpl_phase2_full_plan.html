Reports and analytics
Project-wise revenue report. Employee attendance summary. Leave utilization. Invoice aging report. All exportable as CSV or PDF.

Job seeker portal — 5 modules

Browse jobs public — no login needed
Job listings from the database (same data as careers page). Filter by department, location, experience. Apply button requires login/register. This removes the need to maintain job data in two places.
Register and apply
First-time applicants create a profile (name, email, phone, experience, current role). Then attach resume via Uploadthing. Submit application tied to specific job posting. All stored in JOB_APPLICATION table.
My applications
Track status of all submitted applications: Received → Shortlisted → Interview Scheduled → Hired / Rejected. Notification sent at each stage change by admin.
Interview schedule
When admin schedules an interview, job seeker sees date, time, format (in-person / video), and interviewer name. Can accept or request reschedule (Phase 3 feature — keep it read-only in Phase 2).
Profile
Update personal details, upload updated resume, add work history. This profile is visible to admin when reviewing applications.

Authentication system — single login, role-based redirect

One login page for all users
mmtpl.in/auth/login — single form with email + password. After login, middleware checks the USER.role field and redirects: ADMIN → /admin, EMPLOYEE → /employee, CLIENT → /client, JOB_SEEKER → /jobs. No separate login URLs per role.
User roles (enum in DB)
SUPER_ADMIN (Azghar Ahmed — sees everything, cannot be deleted) · ADMIN (HR, Operations managers) · EMPLOYEE (site engineers, supervisors) · CLIENT (JSW, JSPL contacts) · JOB_SEEKER (applicants)
Auth stack — no NextAuth, use JWT manually
Why: NextAuth adds complexity for this use case. Instead: login API creates a signed JWT (payload: userId, role, exp: 8h) stored in httpOnly cookie. Middleware reads the cookie, verifies JWT, checks role against route. Refresh token in separate httpOnly cookie (exp: 30d).
middleware.ts — the security gatekeeper
Runs on every request to /admin/**, /employee/**, /client/**, /jobs/**. Reads JWT from cookie, decodes it, checks role. If no cookie or wrong role → redirect to /auth/login. All portal API routes also verify the JWT server-side independently (never trust middleware alone for API security).
// middleware.ts — simplified import { NextRequest, NextResponse } from 'next/server' import { jwtVerify } from 'jose' export async function middleware(req: NextRequest) { const token = req.cookies.get('mmtpl_auth')?.value const path = req.nextUrl.pathname if (!token) return NextResponse.redirect(new URL('/auth/login', req.url)) const { payload } = await jwtVerify(token, secret) const role = payload.role as string if (path.startsWith('/admin') && role !== 'ADMIN' && role !== 'SUPER_ADMIN') return NextResponse.redirect(new URL('/auth/login', req.url)) if (path.startsWith('/employee') && role !== 'EMPLOYEE') return NextResponse.redirect(new URL('/auth/login', req.url)) return NextResponse.next() }
